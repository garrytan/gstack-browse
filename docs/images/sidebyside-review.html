## Finding 1 — CRITICAL — N+1 query in process_orders

Location: sample.py:14-18

The loop inside `process_orders` issues two SELECT statements
per user_id. For a batch of 200 users, that is 400 round-trips
to the database. On a hot request path this will dominate
latency and likely trigger alerts on DB connection saturation.

```python
for uid in user_ids:
    user = db.query(User).filter_by(id=uid).first()
    orders = db.query(Order).filter_by(user_id=uid).all()
    ...
```

Recommended fix: batch fetch users with `IN (user_ids)`, then
fetch all orders for those users in a second query and group
in memory by user_id. This reduces the pattern to two queries
total regardless of batch size.

## Finding 2 — CRITICAL — NoneType access on user.tier

Location: sample.py:16

`db.query(User).filter_by(id=uid).first()` returns None if no
such row exists. The subsequent `user.tier` access will raise
AttributeError at runtime, crashing the whole batch on a
single missing user. The existing code before the patch had
the same bug on `user.name`, but the new code adds a second
null-deref site.

Recommended fix: either filter out missing user_ids up front,
or skip with a log line when `user is None`. The right choice
depends on whether a missing user_id is a client bug (should
500) or an expected race (should 200 with a partial response).
Without more context I cannot pick for you, so flag this as a
decision the author needs to make before merge.

## Finding 3 — HIGH — Unhandled KeyError in format_invoice

Location: sample.py:30

```python
symbol = {"USD": "$", "EUR": "€", "GBP": "£"}[currency]
```

Any `currency` value outside the three hard-coded keys will
raise KeyError. The function signature defaults currency to
"USD", which is safe, but any caller that passes a user-supplied
or config-supplied currency value can trigger an unhandled
exception. This is especially bad because format_invoice is
a pure formatter — it should never throw.

Recommended fix: use `.get(currency, "$")` with a sensible
fallback, or validate upstream and return early with a clear
error message if the currency is unsupported.

## Finding 4 — HIGH — refund_order has no null check and no rollback

Location: sample.py:41-44

```python
def refund_order(order_id):
    order = db.query(Order).filter_by(id=order_id).first()
    order.refunded = True
    db.commit()
    return order
```

Two problems. First, `order` can be None — same class of bug
as Finding 2. Second, there is no try/except around the commit,
so a commit failure (constraint violation, connection dropped,
transaction timeout) leaves the caller with no way to
distinguish success from failure. The function silently returns
the stale object reference.

Recommended fix: guard on `order is None` and raise or return
early. Wrap the write in try/except with rollback on failure.
