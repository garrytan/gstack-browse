Benchmark methodology

How CaveStack's savings claim is measured. What tasks we ran. What hardware. How to rerun yourself. Where the numbers are honest, where they are probabilistic, where they can't be audited at all.

v1.0.0.0 — pending first run

CaveStack v1.0.0.0 ships with the benchmark harness but not yet with numbers. The maintainer runs bun run bench once before tagging the release, which populates docs/benchmarks/v1.0.0.0.json and updates the hero counter above. Until then the site shows see benchmarks with a link here.

The task set

Ten fixed software engineering tasks spanning common Claude Code workloads. Each task has a well-defined success criterion. The full list lives at test/benchmarks/tasks.md.

  • feature — dark-mode toggle, migration, etc.
  • debug — investigate 500 error, debug flaky test
  • refactor — rename function across files
  • review — SQL injection audit
  • deploy — ship feature branch
  • security — OWASP Top 10 audit
  • design — pricing page review
  • tests — billing module coverage

Tasks change only between major versions. Every version freezes its own task set, so v1.0.0.0 always reproduces with the same inputs.

Frameworks measured

  1. Raw Claude Code — vanilla install, default settings, no CaveStack.
  2. CaveStack — current version installed normally.
  3. SuperClaude — included only if SUPERCLAUDE_DIR is set during the benchmark run AND SuperClaude's license permits publication. If license blocks, column is null and hero copy adjusts accordingly.

Hardware + environment

  • Claude Opus 4.7 (pinned model snapshot per release)
  • Anthropic API with maintainer-owned key (not user-paid)
  • macOS 14.5, M3 MacBook Pro, 32GB RAM (reference; actual recorded in results JSON)
  • CaveStack installed globally, cold session per task
  • Residential fiber, April 2026

Re-run it yourself

Every number is reproducible. Clone the repo, install CaveStack, run the harness with your own API key:

$git clone https://github.com/JerkyJesse/cavestack
$cd cavestack && ./setup
$export ANTHROPIC_API_KEY="sk-ant-..."
$bun run bench
$cat docs/benchmarks/v1.0.0.0.json

Your numbers will differ slightly. Post the diff on Issues if it deviates more than 15% from the published results.

Three trust tiers

This matters. Different claims on this site come with different levels of verifiability. We label them explicitly instead of pretending they are all the same.

Verifiable Benchmark task set, harness code, per-task token counts, reproduction steps. Anyone with an API key can rerun and compare.
Probabilistic The "typical user saves X" framing. Your workload differs from our task set. Real savings depend on what YOU ask Claude Code to do. We publish the task distribution so you can judge fit.
Unaudited Anecdotes, social media screenshots, individual users' claims. Useful signal, not proof. Treat as "someone said this" not "measured fact."

What this does NOT prove

Task set is maintainer-selected. Different tasks produce different ratios. We pick tasks we consider representative, not tasks that maximize CaveStack's advantage. If your workload differs, your savings differ. Add tasks via PR; we re-run on the next release.
Model revisions shift baselines. Anthropic updates Opus periodically. We pin the benchmark to a specific model snapshot and document it in each results JSON. A later release might show different numbers because the model changed, not because CaveStack did.
No continuous monitoring. We do not run weekly cron benchmarks on user machines or maintainer machines. Results are taken once per release. If a later CaveStack release regresses savings, the next benchmark catches it; we don't see it instantly.

Zero telemetry promise

CaveStack never phones home. The benchmark runs only on the maintainer's machine, once per release. Your install does not participate. The numbers on this page come from controlled runs, not aggregated user data. If you want to verify, the full harness output is downloadable and the harness code is in the public repo.
